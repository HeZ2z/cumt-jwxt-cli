# CLAUDE.md

## 项目定位

`cumt-jwxt-cli` 是面向 CUMT 教务系统的 Python CLI。首版只实现成绩查询、成绩变更检测和邮件通知；旧项目仅作为需求和接口参考。

## 常用命令

```bash
uv sync
uv run cumt-jwxt --help
uv run cumt-jwxt grades query --help
uv run pytest
uv run ruff check .
```

## 工作方式

- 先阅读相关文件再修改，不基于猜测改代码。
- 优先最小 diff，避免无关重构。
- 新增或修改业务逻辑时同步更新测试。
- 改完运行相关测试和 `uv run ruff check .`。
- 涉及多模块或不确定需求时，先给方案再实施。
- 任何时候都不得记录、提交或输出真实账号、密码、cookie、session、API Key、验证码图片或敏感响应。

## 注意事项

- 使用 `uv`，不要用 `pip install`、Poetry 或 Pipenv 修改依赖。
- `config.example.json` 只作模板，不作为默认运行配置读取。
- `config.local.json` 和 `config.json` 都是本地真实配置，不提交。
- 单元测试不得访问真实教务系统、模型服务或 SMTP。
- 验证码图片默认不落盘，任何情况下不得提交。

## 安全注意事项

不得记录或提交以下内容：

- 账号、密码、API Key
- cookie、session、JSESSIONID、route
- 验证码图片
- HAR 文件或含敏感响应的调试文件
- 日志、输出、临时文件和其他运行产物

实现日志、错误信息和测试 fixture 时必须避免泄露真实敏感信息。

## Python 工具链约定

- 使用 `uv` 管理依赖、虚拟环境和锁文件，不使用 `pip install`、`poetry` 或 `pipenv` 修改项目依赖。
- 新增运行时依赖使用 `uv add <package>`；新增开发依赖使用 `uv add --dev <package>`。
- 修改依赖后必须提交 `pyproject.toml` 和 `uv.lock`。
- 优先使用标准库；新增第三方依赖前需要确认其必要性。
- 项目支持 Python 3.11+。
- 开发环境可以使用更新版本 Python，但代码必须保持 Python 3.11 兼容。
- 不使用 Python 3.12+ 专属语法或标准库 API，除非先同步调整 `requires-python`、测试矩阵和文档。

## 重要目录

- `src/cumt_jwxt_cli/cli.py`: CLI 参数解析和命令入口。
- `src/cumt_jwxt_cli/config.py`: 配置加载。
- `src/cumt_jwxt_cli/client/`: 登录、会话和教务系统 HTTP 交互。
- `src/cumt_jwxt_cli/captcha/`: 验证码识别或人工输入。
- `src/cumt_jwxt_cli/grades/`: 成绩查询、解析、比较、快照和报告。
- `src/cumt_jwxt_cli/notify/`: 邮件通知。
- `tests/`: pytest 测试。

## 结构约定

- 使用 `src/cumt_jwxt_cli` 包结构。
- CLI 入口为 `cumt_jwxt_cli.cli:main`，命令名为 `cumt-jwxt`。
- 首版命令范围只做 `cumt-jwxt grades query`。
- 不迁移旧项目 `xuanke`，不要保留旧入口 `app.py`、`app.sh`、`xuanke.sh`。

## 测试约定

- 使用 `pytest` 编写测试，测试目录为 `tests/`。
- 新增或修改业务逻辑时必须同步补充或更新测试。
- 网络请求、验证码识别、邮件发送等外部交互必须 mock，不在单元测试中访问真实教务系统、真实模型服务或真实 SMTP。
- 测试 fixture 不得包含真实账号、密码、cookie、验证码图片或教务系统敏感响应。
- 修改 CLI 行为时至少覆盖 `--help`、参数校验和主要错误路径。

## 代码风格约定

- 使用 Ruff 进行 lint 和格式化。
- 提交前至少运行：
  - `uv run ruff check .`
  - `uv run pytest`
- 修改较多 Python 代码后运行：
  - `uv run ruff format .`
  - `uv run ruff check . --fix`
- 不引入 Black、isort、flake8 等额外格式化/lint 工具，除非先调整 `pyproject.toml` 和项目约定。
- 遵循 `pyproject.toml` 中的 Ruff 配置：行宽 88，目标版本 py311。

## 类型标注约定

- 新增或修改的函数应补充参数和返回值类型标注。
- 业务模型优先使用 `dataclasses` 或明确的类型结构，不用松散的 `dict[str, Any]` 在模块间传递核心数据。
- 仅在边界层解析外部 JSON、HTML、配置时使用宽泛类型；进入业务层前转换为明确模型。
- 暂不引入 mypy/pyright，除非项目复杂度明显上升或测试无法覆盖类型风险。

## 注释与文档字符串约定

- 优先通过清晰命名、短函数和类型标注表达意图，不为显而易见的代码添加注释。
- 注释重点解释“为什么这样做”、教务系统接口的非显然行为、兼容性限制和安全边界。
- 不添加复述代码行为的注释，例如“遍历列表”“发送请求”“返回结果”。
- 公共模块、公共类和复杂业务函数应添加简洁 docstring；内部小函数仅在逻辑不直观时添加 docstring。
- 涉及敏感信息处理、日志脱敏、状态文件写入、验证码处理、邮件发送时，应在必要位置说明安全原因。
- 不在注释、docstring、测试或示例中写入真实账号、密码、cookie、session、API Key、验证码内容或真实敏感响应。
- 保持注释随代码同步更新；删除过期注释，不保留“旧逻辑”“临时方案”“以后再改”等无效说明。

## CLI 设计约定

- CLI 输出默认面向终端用户，保持简洁、可读，不输出调试细节。
- 成功结果输出到 stdout；错误信息输出到 stderr。
- 使用稳定退出码：成功为 0，用户输入或配置错误为 2，外部服务/网络/认证失败为 1。
- 命令行参数优先级高于配置文件；未提供参数时读取配置文件默认值。
- 默认不进行会阻塞自动化流程的交互式输入；如必须交互，应提供参数或配置绕过方式。
- `--help` 文案应与 README 和 `config.example.json` 保持一致。

## 配置读取约定

- 仓库提交 `config.example.json`，本地真实配置优先使用 `config.local.json`。
- 不提交 `config.local.json` 或 `config.json`。
- 可通过 `--config` 显式指定配置文件路径。
- 未指定 `--config` 时，按顺序查找项目根目录或当前工作目录下的 `config.local.json`、`config.json`。
- `config.example.json` 只作为模板，永远不作为默认运行配置读取。
- 配置优先级从高到低为：命令行参数、环境变量、配置文件、代码默认值。
- 必需配置缺失时应给出明确错误信息，不静默使用空字符串继续请求外部服务。
- 密码、API Key 等敏感字段允许通过环境变量提供，优先于配置文件。
- 运行状态写入实际配置文件旁的 `state.json`，不要写回配置文件。

## 成绩数据约定

- 成绩解析后应转换为明确的数据模型，再进行展示、比较、保存或通知。
- 变更检测应基于稳定字段生成摘要或哈希，避免因字段顺序、空白字符、HTML 结构变化导致误报。
- 成绩记录应尽量保留课程号、课程名、学年、学期、成绩、绩点、学分等可用字段。
- 对缺失字段应显式建模为 `None` 或空值，不用魔法字符串表示。
- 展示层格式化不得反向作为状态比较依据。

## 分层依赖约定

- `cli` 只负责参数解析、输出和流程编排，不直接解析 HTML 或发送底层 HTTP 请求。
- `client` 负责登录、会话和教务系统 HTTP 交互。
- `captcha` 负责验证码识别或人工输入，不持久化验证码。
- `grades` 负责成绩查询、解析、比较、快照和报告生成。
- `notify` 负责通知发送，不承担成绩比较逻辑。
- 底层模块不得依赖 CLI；业务逻辑应可被测试直接调用。

## 错误处理约定

- 为配置错误、认证失败、验证码失败、网络失败、解析失败、通知失败使用清晰的项目内异常类型。
- 边界层捕获外部库异常并转换为项目内异常，避免在业务层扩散 httpx、SMTP、OpenAI SDK 等底层异常。
- CLI 层负责把项目内异常转换为用户可读错误和稳定退出码。
- 不使用裸 `except Exception` 吞掉错误；需要兜底时必须保留安全、可诊断的信息。

## HTTP 与认证约定

- 使用受控的 HTTP client/session 管理 cookie，不在模块间传递裸 cookie 字符串。
- 不把 cookie、JSESSIONID、route 等会话信息写入日志、状态文件、报告或测试 fixture。
- 登录态只在单次运行内保留，默认不持久化。
- 认证失败应明确报错，不自动无限重试登录。
- 所有请求应集中设置 User-Agent、超时和重试策略，避免各模块重复实现。

## HTML 解析约定

- 使用 BeautifulSoup 解析教务系统页面，不用脆弱的正则表达式解析复杂 HTML。
- 解析逻辑应容忍无关空白、字段顺序和少量结构变化。
- 对关键字段缺失应返回明确解析错误，不静默生成错误成绩。
- 解析测试应使用脱敏、最小化的 HTML fixture，不保存真实完整响应。

## 环境变量约定

- 项目环境变量统一使用 `CUMT_JWXT_` 前缀。
- 敏感字段可通过环境变量覆盖配置文件，例如账号、密码、验证码服务 API Key、SMTP 密码。
- README、`config.example.json` 和 CLI help 中出现的环境变量名称应保持一致。
- 不读取无前缀的通用变量名，避免与用户系统环境冲突。

## 并发与频率约定

- 对教务系统请求保持保守并发，默认详情查询并发不超过配置中的 `detail_concurrency`。
- 不为提速引入无上限并发。
- 重试和并发设计应避免对教务系统造成明显压力。
- 并发失败时应能定位到具体课程或请求阶段，但不得泄露敏感响应。

## 报告生成约定

- 报告只基于已解析的成绩模型生成，不直接嵌入原始 HTML 响应。
- 生成 HTML 时必须对课程名、教师名等外部文本进行转义，避免 HTML 注入。
- 报告中不得包含账号、cookie、session、API Key 或完整原始响应。
- 报告保存必须由用户显式启用。

## 自动化运行约定

- 命令应支持非交互运行，适合 cron、launchd、CI 或其他自动化调度。
- `--no-interactive` 模式下不得等待用户输入；缺失配置、验证码或认证失败应快速失败。
- 自动化运行时不得默认生成大量日志或产物。
- 通知发送失败应有明确退出状态或警告，便于调度系统发现问题。

## 邮件通知约定

- 邮件通知默认关闭，只有 `notify.enabled` 或显式参数启用时才发送。
- 只在检测到成绩变化或用户显式要求测试通知时发送邮件。
- 邮件内容不得包含账号、密码、cookie、session、API Key 或完整敏感响应。
- SMTP 密码等敏感字段允许通过环境变量提供，优先于配置文件。
- 邮件发送失败不应破坏已完成的成绩查询结果，但应返回清晰错误或警告。
- 测试中必须 mock SMTP，不发送真实邮件。

## 验证码处理约定

- 验证码图片只在内存中处理，默认不落盘。
- 自动识别失败时可回退到人工输入，但必须受超时控制，避免自动化任务无限阻塞。
- 验证码识别服务的 API Key、base_url、模型名从配置或环境变量读取，不写入代码。
- 不在日志、测试 fixture、报告或错误信息中保存验证码图片、识别请求内容或敏感响应。
- 测试验证码流程时使用 mock 或人工构造的非真实样例，不使用真实验证码图片。

## 运行产物约定

- 默认不保存 JSON 成绩、HTML 报告、验证码图片、原始响应或调试文件。
- 只有用户显式通过配置或 CLI 参数启用时，才保存 JSON 成绩或 HTML 报告。
- 即使启用输出，也不得保存验证码图片、cookie、session、HAR 或完整敏感响应。
- 输出文件应写入显式指定的输出目录；未指定时使用配置文件同目录下的受控输出目录。
- 生成的报告应避免包含账号、cookie、session、API Key 等敏感信息。

## 状态文件约定

- `state.json` 仅保存成绩变更检测所需的最小状态，例如上次查询摘要、成绩哈希、通知时间等。
- 不在 `state.json` 中保存账号、密码、API Key、cookie、session、验证码图片或完整敏感响应。
- 状态文件结构变更时应保持简单可迁移；首版不引入数据库。
- 状态写入应尽量采用原子写入，避免中途中断导致文件损坏。
- 状态读取失败时应给出清晰错误；不要静默覆盖无法解析的既有状态文件。

## 外部服务调用约定

- 所有外部 HTTP 请求必须设置超时，不允许无限等待。
- 网络瞬时错误可按配置进行有限重试；认证失败、参数错误、验证码错误不应盲目重试。
- 重试必须有退避间隔，避免对教务系统或第三方服务造成压力。
- 不在测试中访问真实教务系统、真实模型服务或真实 SMTP。
- 外部服务响应只在边界层解析，进入业务层前转换为明确模型或结果对象。
- 对教务系统接口的特殊行为，应在代码注释或测试名称中记录原因。

## 日志与错误信息约定

- 日志只能记录流程状态、脱敏后的标识和可诊断的非敏感错误信息。
- 不记录请求头、cookie、session、验证码图片、账号、密码、API Key、完整请求/响应正文。
- 对外部异常进行包装时，不直接透传可能包含敏感信息的原始响应内容。
- CLI 错误输出应面向用户可理解，避免打印 traceback，除非显式启用 debug 模式。
- 如后续支持 debug 日志，必须默认关闭，并继续遵守敏感信息脱敏规则。

## 完成标准

任务完成前应确认：

- 代码修改符合现有分层和安全约定。
- 新增或修改业务逻辑已有对应测试。
- 已运行相关测试；提交前至少运行 `uv run ruff check .` 和 `uv run pytest`。
- 未写入或提交真实配置、运行状态、日志、验证码、cookie、session 或其他敏感产物。
- README、`config.example.json`、CLI help 与实际行为保持一致。
